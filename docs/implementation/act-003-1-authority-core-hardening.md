# ACT-003.1 Authority-Core Hardening

ACT-003.1 corrects the bindings that make clearance channel policy enforceable.
It is a correction sprint, not a feature expansion.

## What Changed

- Clearance channel now comes from the authenticated principal record
  (`mobile_signed` or `local_terminal`), not from the route handler.
- ACT-003.1.1 moves that class source to pairing initiation. The operator pins
  `clearance_channel` on the pairing session; pairing completion uses the
  session value and rejects a conflicting device-supplied class.
- New principal creation defaults to `local_terminal`, not `mobile_signed`.
- Existing explicit device rows are not rewritten.
- Local terminal decisions require canonical Ed25519 request signatures from a
  registered `local_terminal` principal.
- The old local-terminal `signature_verified` boolean and free-form
  `terminal_identity` body fields are removed.
- Channel eligibility is enforced only for approvals/grants. Deny, expire, and
  cancel are authority-reducing and may proceed from any authenticated channel.
- New agents default to `untrusted_host`.
- An operator-only signed endpoint sets per-aircraft deployment trust context:
  `PATCH /v1/agents/{agent_id}/deployment-trust-context?node_id=...`.
- `risk_family` is required on clearance requests. No unlabeled request defaults
  to a low-risk family.
- `params_fingerprint` is computed over the redacted action payload, returned on
  approvals, and must be echoed in scoped signed approval decisions.
- External request schemas reject tower-owned `deployment_trust_context` and
  `channel_eligibility` fields.

## Migration Note

The schema default for new agent rows is `untrusted_host`. Existing explicit
trust context values are preserved so operator-set `trusted_host` rows are not
silently rewritten on restart. Operators should review existing aircraft and
set trust context intentionally through the signed operator endpoint.

The schema default for new principal `clearance_channel` values is
`local_terminal`. Existing device rows keep their explicit channel class,
including legitimately paired `mobile_signed` phones.

## Honest Limit

ACT-003.1.1 pins channel class at operator-mediated pairing initiation and
removes device-declared class selection. It still does not prove that a
`mobile_signed` key was generated by Secure Enclave or Android Keystore.
Hardware-attested binding remains code-complete-only until validated on real
devices.

## Remaining Risk

The backend-provided `risk_family` label is still trusted for routing. A future
capability registry should pin risk family per known backend capability.
